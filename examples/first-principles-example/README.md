## Phase 1: 6 路并行信息采集

6 个 Agent 同时从不同维度搜索和聚合信息。

### Agent 1: 经典文献
📄 [`phase1-research/01-classic-literature.md`](phase1-research/01-classic-literature.md)

**核心发现**：
- Aristotle（公元前350年）：定义"第一原理"为知识的基础公理
- Descartes（1641年）：系统化怀疑方法，区分已知与假设
- Musk（2012年）：工程化第一性原理，提出类比推理 vs 第一性原理二分法

### Agent 2: 实践案例
📄 [`phase1-research/02-practice-cases.md`](phase1-research/02-practice-cases.md)

**核心发现**：
- **成功**：SpaceX 火箭（从材料成本推导自制可行性）、Tesla 电池（从原材料现货价推导成本）、Amazon 电商（从用户基本需求推导）
- **失败**：Theranos（基本事实被歪曲）、Juicero（过度工程化忽略真实需求）

### Agent 3: 批判视角
📄 [`phase1-research/03-critical-views.md`](phase1-research/03-critical-views.md)

**核心发现**：
- 时间成本高：不适合日常决策
- 还原论局限：复杂系统不可简单还原
- 基本事实可靠性："公认事实"可能被后来的科学突破推翻
- 幸存者偏差：Musk 的成功不一定归因于第一性原理本身

### Agent 4: 跨域迁移
📄 [`phase1-research/04-cross-domain.md`](phase1-research/04-cross-domain.md)

**核心发现**：
- 物理学：第一性原理的原生地（Newton、Einstein）
- 法学：宪法解释和法理分析
- 计算机科学：算法设计和 Bitcoin
- 医学：鉴别诊断（Differential Diagnosis）
- 烹饪：分子料理

**跨域共性模式**：基本公理推演 → 机制分析优先 → 类比检查意识 → 证据驱动验证

### Agent 5: 工具化程度
📄 [`phase1-research/05-tooling.md`](phase1-research/05-tooling.md)

**核心发现**：
- **已有框架**：Musk 3 步法（过简）、Descartes 系统化怀疑（过重）、5 Why（不系统）
- **工具缺口**：缺少完整的可操作协议 → forge-skill 的机会点
- **评估**：工具化程度低，主要停留在概念框架层面

### Agent 6: 演化时间线
📄 [`phase1-research/06-timeline.md`](phase1-research/06-timeline.md)

**核心发现**：
```
Aristotle(350BCE) → Descartes(1641) → Popper(1934) → Musk(2012) → 当前
  哲学起源            系统化              科学方法          工程化          工具化(目标)
```

**关键洞察**：第一性原理经历了从本体论→方法论→工程化的演化，但尚未完成"工具化"的最后一步。forge-skill 正在完成这一步。

## Phase 2: 方法论提炼

### 2.1 核心原理提取

**三重验证过程**：

| 候选原理 | 跨源验证（≥2） | 跨域验证（≥2） | 实践验证 | 通过 |
|---------|----------------|----------------|---------|------|
| 还原到基本事实 | Aristotle + Descartes + Musk (✓3) | 物理 + 航天 + 汽车 (✓3) | SpaceX + Tesla (✓2) | ✓ 原理1 |
| 区分已知与假设 | Descartes + Popper (✓2) | 科学方法 + 工程 (✓2) | 鉴别诊断 (✓1) | ✓ 原理2 |
| 从基本事实向上推导 | Musk + Bezos (✓2) | 物理 + 商业 (✓2) | SpaceX + Amazon (✓2) | ✓ 原理3 |
| 警惕类比陷阱 | Musk + Kuhn (✓2) | 科学哲学 + 产品设计 (✓2) | iPhone + Juicero (✓2) | ✓ 原理4 |
| 可证伪性 | 仅 Popper | 不足 | 不足 | ✗ 合并到原理2 |

**最终选择**：4 个核心原理（满足 3-7 个的要求）

### 2.2 操作流程构建

基于 4 个核心原理构建 3 步操作协议：

**Step 1: 问题分类**
- 适用信号：行业惯例效果不好、需要根本性选择、技术创新挑战
- 不适用信号：情感/人际关系问题、快速执行场景、信息不足

**Step 2: 第一性原理式拆解**（4 个研究维度对应 4 个原理）
1. 基本约束分析 ← 原理 1（还原到基本事实）
2. 当前假设拆解 ← 原理 2（区分已知与假设）
3. 最小可行单元识别 ← 原理 3（从基本事实向上推导）
4. 类比检查 ← 原理 4（警惕类比推理陷阱）

**Step 3: 第一性原理式输出**
- 基本事实清单
- 当前方案假设及验证状态
- 基于基本事实的替代方案
- 置信度和下一步验证

### 2.3 适用边界分析

| 问题类型 | 适用度 | 说明 |
|---------|--------|------|
| 技术创新方案选择 | 高 | 最适合需突破现有技术范式的场景 |
| 商业策略重新定义 | 高 | 适合挑战行业惯例的业务模式创新 |
| 个人生涯关键决策 | 中 | 有启发但需结合价值观 |
| 日常运营优化 | 低 | 用改善（Kaizen）更合适 |
| 情感/人际问题 | 低 | 第一性原理处理逻辑问题 |

### 2.4 诚实边界

1. **不可还原的复杂系统**：涌现现象不能通过还原完全解释
2. **物理极限本身可能是错误的**：历史上多次被认为的极限后来被突破
3. **时间成本高**：不适合每天使用，仅在关键决策时启用
4. **信息不对称风险**：不了解领域基本事实会导致 Garbage-in-Garbage-out

## Phase 3: Skill 构建

基于 Phase 2 的提炼结果，按 forge-skill 模板构建完整的 `SKILL.md`。

**构建的 Skill 结构**：

```
methods/first-principles-skill/SKILL.md
├── YAML frontmatter (name, description, triggers)
├── 激活条件与触发词
├── 方法框架概览
├── 核心原理 (4个，各附 ≥2 跨域证据)
│   ├── 原理 1: 还原到基本事实
│   ├── 原理 2: 区分已知与假设
│   ├── 原理 3: 从基本事实向上推导
│   └── 原理 4: 警惕类比推理陷阱
├── 操作协议 (3 Steps)
├── 适用/不适用判断 (5 种场景)
├── 典型案例库 (3 成功 + 1 失败)
├── 误用检测器 (3 种误用模式)
├── 诚实边界 (4 条具体局限)
└── 调研来源 (8 个来源，75% 一手)
```

### 从"概念"到"工具"的转变

锻造前，第一性原理主要停留在：
- Musk 的 3 步法（简化概念）
- 学术界的方法论讨论
- 碎片化的案例分享

锻造后，第一性原理成为一个完整的可操作协议：
- **可判断**：Step 1 告诉你什么时候该用、什么时候不该用
- **可执行**：Step 2 的 4 个研究维度提供了具体的分析步骤
- **可防御**：诚实边界和误用检测器帮你识别常见陷阱
- **可溯源**：每个原理都连接到一手来源
